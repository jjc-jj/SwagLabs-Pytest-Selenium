## 九、重要踩坑记录（面试可讲）

### 坑1：禅道自动化提交连续失败6轮（70分钟）
- **问题**：Selenium 连续尝试6种方案，表单始终不渲染（ZenTao 6.3 是 Zin/SPA 框架，picker 组件与 WebDriver 不兼容）
- **转折**：放弃脚本猜测，使用 **F12 Network 抓包** 手动操作一次，捕获真实 API 请求
- **解决**：`POST /api.php/v1/bugs` REST API，JSON payload，3分钟完成
- **教训**：**先看 Network，再写代码**——F12 手动操作一次抓到的真实 HTTP 请求比任何文档都准确

### 坑2：API 提交到错误项目
- **问题**：从表单 HTML 模板中提取的 `productID=8, projectID=24` 是默认占位值，非用户实际项目
- **解决**：以 F12 抓到的真实 POST 数据为准（`product=73, project=162`）

### 坑3：Excel 合并单元格导致 openpyxl 写入失败
- **问题**：openpyxl 对 MergedCell 写入抛出 AttributeError
- **解决**：v2.0 重构时从零构建工作簿，数据区完全扁平化

### 坑4：安全测试判定标准偏差
- **问题**：SQL注入/XSS 返回通用错误消息，预期结果写成"显示安全拦截"，导致误判Fail
- **解决**：AI审计指出不泄露防护细节的通用错误消息才是安全最佳实践

## 十、F12 Network 抓包方法论（核心经验）

> **核心原则：先看 Network，再写代码**

### 操作流程
1. F12 → Network → 勾选 **Preserve log**
2. 手动执行一次完整操作 → 触发真实 HTTP 请求
3. 筛选 **XHR/Fetch** 类型 → 过滤 JS/CSS/图片噪音
4. 查看 **Payload** 标签 → 获取请求体字段名、类型、示例值
5. 查看 **Response** 标签 → 获取响应格式和错误提示
6. 右键 → **Copy as cURL** → 直接获得可复用的请求模板

### 适用场景
Bug提交、接口测试、数据抓取、任何需要自动化操作 Web 系统的任务

## 十一、企业级重构标准（后续操作参考）

### 测试用例标准
- 编号格式：`TC-<模块缩写>-<3位序号>`（如 TC-LOGIN-001）
- 必填字段：用例ID、所属模块、用例标题、优先级(P0-P3)、前置条件、测试步骤、预期结果、实际结果、执行状态
- 标题规范：`验证+操作对象+预期结果`
- 步骤规范：`1. 2. 3.` 编号，每步含具体测试数据
- 预期结果：可验证的客观描述

### 缺陷报告标准
- 缺陷ID：`DEF-<3位序号>`
- 必填字段：关联用例、标题、模块、严重程度(致命/严重/一般/轻微)、优先级(P0-P3)、类型、环境、前置条件、复现步骤、预期结果、实际结果、影响范围、修复方案、状态
- 复现步骤：清晰的 1,2,3 步骤，含具体输入数据

### 格式规范
- ❌ 禁止合并单元格（数据区）
- ✅ 扁平化数据，每行一条独立记录
- ✅ 严谨客观的工程语言
