### 🔬 F12 Network 抓包方法论总结（面试可讲）

> **背景**：禅道Bug自动提交任务中，连续6轮Selenium脚本尝试全部失败（表单不渲染/提交被拒/CSRF失败），累计耗时70分钟无产出。
>
> **转折点**：放弃脚本猜测，打开浏览器 **F12 → Network 面板**，手动提交一个Bug，观察XHR请求：
> ```
> POST /api.php/v1/bugs  →  201 Created  →  {"id": 75}
> Content-Type: application/json
> {"title":"...","product":73,"project":162,"severity":2,...}
> ```
> 得到完整的 **API端点 + 请求格式 + 字段值 + 认证方式**，5分钟内完成全部11条Bug提交。
>
> **方法论提炼**：
>
> | 步骤 | 操作 | 获得信息 |
> |------|------|---------|
> | 1 | F12 → Network → 勾选 Preserve log | 保留完整请求历史 |
> | 2 | 手动执行一次完整操作 | 触发真实的HTTP请求 |
> | 3 | 筛选 XHR/Fetch 类型 | 定位API调用（过滤JS/CSS/图片噪音）|
> | 4 | 查看 Payload 标签 | 获取请求体字段名、类型、示例值 |
> | 5 | 查看 Response 标签 | 获取响应格式和错误提示 |
> | 6 | 右键 → Copy as cURL | 直接获得可复用的请求模板 |
>
> **核心原则**：
> > **先看 Network，再写代码。** 不要猜测后端接口——F12手动操作一次抓到的真实HTTP请求，比任何文档和推理都准确。每条cURL命令都是一份可执行的API文档。
>
> **适用场景**：任何需要自动化操作Web系统的任务（Bug提交、接口测试、数据抓取），此方法论可普遍适用。

### 🔬 踩坑与重构故事（面试必讲）

#### 故事一：pytest-html 报告表格空白（70 分钟）

> **遇到的坑**：测试全部通过，pytest-html 报告顶部统计栏正常显示"11 tests, 3 Failed, 8 Passed"，但下方详情表格完全空白，"Show all details"按钮点击无响应。
>
> **排查过程**：
> 1. 第一轮猜测：JSON 数据损坏 → Python 解析验证 → JSON 完整无损
> 2. 第二轮猜测：数据量过大导致 JS 渲染超时 → 加 --capture=no → 仍然空白
> 3. 第三轮猜测：--self-contained-html 内联问题 → 去掉参数 → 仍然空白
> 4. 第四轮（关键）：打开 F12 Console → 发现 `SecurityError: Failed to execute 'pushState' on 'History'`
>
> **根因定位**：pytest-html 的 JS 代码使用 `window.history.pushState()` 管理排序状态，Chromium 在 `file://` 协议下禁止此 API（origin=null），JS 初始化崩溃，表格未渲染、事件监听器未绑定。
>
> **尝试过的方案（被否决）**：
> - 启动 python -m http.server 通过 localhost 访问 → 可行但每次手动操作太麻烦
> - 降级 pytest 到 8.x → 会丢失其他依赖的兼容性
> - 换用 Allure → 需额外安装 CLI 工具，GitHub 被墙无法下载
>
> **最终解决方案**：在 conftest.py 的 `pytest_unconfigure` hook（trylast=True，确保在 pytest-html 写报告之后执行）中自动修改 report.html，将 `window.history.pushState(...)` 包裹为 `try{...}catch(e){}`。每次测试结束后全自动修复。
>
> **量化效果**：
> - 修复前：报告双击打开 → 表格空白 → 需要手动启动 http.server
> - 修复后：报告双击打开 → 表格正常渲染 → 零额外操作
> - 适用面：Chrome/Edge/Opera/Brave 所有 Chromium 内核浏览器

#### 故事二：Sauce Demo 跨境 CDN 超时

> **遇到的坑**：Edge 浏览器手动访问 Sauce Demo 秒开，但 Selenium 自动化 `driver.get()` 频繁超时（26-30s），报 `TimeoutException: Timed out receiving message from renderer`。
>
> **排查过程**：观察日志发现页面加载阶段卡在"等待渲染器响应"——手动秒开是因为浏览器缓存了 CSS/字体/图片，Selenium 每次新开无缓存实例需要重新下载全部 CDN 资源。
>
> **尝试过的方案**：
> - time.sleep(30) → 治标不治本，且违反零 sleep 原则
> - 换 Chrome → 同样 CDN 问题
> - 配置代理 → 境外 CDN 走代理可能更慢
>
> **最终解决方案**（三层优化）：
> 1. `page_load_strategy='eager'` — DOM 就绪即继续，不等待图片/字体
> 2. `--disable-features=DnsOverHttpsUpgrade` — 禁用 DoH 避免 DNS 解析超时
> 3. `profile.managed_default_content_settings.images: 2` — 浏览器级拦截图片请求（可配置开关）
>
> **量化效果**：
> - 修复前：页面加载 12-30s，约 30% 概率超时
> - 修复后：页面加载 2-5s，超时率降至 5% 以下
> - 每日执行节省时间：约 60%（全量 11 条从 267s 降至 110s 以下）

#### 故事三：Edge/Chrome 驱动选择策略

> **决策背景**：ChromeDriver 托管在 Google 服务器（国内被墙），每次下载驱动都失败。如果每次让用户手动下载驱动放到 PATH 里，团队协作和 CI/CD 都无法推广。
>
> **决策**：Edge + EdgeDriver 替代 Chrome + ChromeDriver。EdgeDriver 托管在微软服务器（国内直连），使用 webdriver-manager 自动下载匹配版本。
>
> **收益**：
> - 零翻墙依赖：微软 CDN 国内直连，下载速度 5-10MB/s
> - CI/CD 友好：GitHub Actions ubuntu-latest 可直接使用（虽然默认无 Edge，但可通过 sed 切换回 Chrome + 安装 Chrome）
> - 面试加分：体现工程化思维——"解决问题的最佳方式有时不是攻克障碍，而是绕过障碍"
